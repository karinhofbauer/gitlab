Git - rebase and merge

Rebase a branch with the master

Switch to the master branch, add a new file and commit it.

$ echo Our fourth file > fourthfile
$ git add fourthfile
$ git commit -m ”Added fourth file.”
(1)-(2)-(3)-(4)-(7*) -- master**
              \
             (5)-(6*) -- newbranch

Switch to the newbranch.

We will now rebase on master.

$ git rebase master
(1)-(2)-(3)-(4)-(7*) -- master
                  \
                 (5)-(6*) -- newbranch**

What happended? Commit 5 and 6 have been reversed and tucked away, commit 7 has been applied and afterwards commit 5 and 6 got re-applied. No conflicts occured in these commits.

The newbranch can now safely be pushed on to the master, since they share the same root.

$ git checkout master $ git push . newbranch:master
(1)-(2)-(3)-(4)-(7)-(5)-(6*) -- master
                           \
                           newbranch**

Once a topic branch has been merged into master, it can be discarded. Switch to the master. Delete the newbranch.

$ git branch -d newbranch

Verify that there is only one branch left, the master.

Since branching in git is so easy it is a good way to go to use many topic branches, one for each feature you're working on.

(1)-(2)-(3)-(4)-(7)-(5)-(6*) -- master**

Handling conflicts

Now create a new branch, named otherbranch. Stay on the master, add a line to the firstfile and commit.

$ echo This is in master. >> firstfile
$ git commit -a -m “Added master line”
(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8*) -- master**
                           \
                           otherbranch

Switch to the otherbranch and add a line to the firstfile there aswell.

$ echo This is in otherbranch. >> firstfile
$ git commit -a -m “Added otherbranch line”
(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8*) -- master
                           \
                           (9*) --otherbranch**

Now if this isn't the perfect setup for a conflict! And since we want to have some fun we just try to rebase on master.

$ git rebase master

Oops, conflict! Completely unexpected...

Check the status and notice that we're not on any branch. We're in rebase limbo, looking at commit #9.

Lets solve this manually. Bring up firstfile in your favourite text editor. Note the merge markers. We want to keep both lines, so remove the merge markers and keep the lines. Now add the file back to the commit and continue with the rebase.

$ git add firstfile
$ git rebase --continue
(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8*) -- master
                               \
                               (9*) --otherbranch**

Switch to master and try to delete the branch.

$ git checkout master
$ git branch -d otherbranch

We're normally not allowed until it has been pushed to master. Let's do that. Be sure to switch to it first.

$ git checkout otherbranch
$ git push . otherbranch:master
$ git checkout master
$ git branch -d otherbranch
(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8*)-(9*) -- master**

Merging

Rebase will alter the history, but guarantee a linear tree. We can however also merge. Lets set up a branch.

$ git checkout -b mergebranch

Add a fifthfile to mergebranch and commit.

$ echo ′A line to merge.′ > fifthfile
$ git add fifthfile && git commit -a
(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8)-(9*) -- master
                                  \
                                  (10*) --mergebranch**

Do some changes to master and commit.

$ git checkout master
$ echo "If this file gets bigger than one MB it will destroy itself and take your computer with it!" >> firstfile
$ git commit -a -m ”The usual nonsense.”
(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8)-(9)-(11*) -- master**
                                  \
                                  (10*) --mergebranch

You can inspect all branches with gitk.

$ gitk --all

Verify that master is your current branch. Now we merge mergebranch into master, preserving the commit order.

$ git merge mergebranch.
(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8)-(9)-(11)-(12*) -- master**
                                   \    /
                                   (10*) --mergebranch

Note that the merging resulted in its own commit.

Switch to mergebranch add a line to the firstfile and commit.

$ git checkout mergebranch
$ echo Conflict ahoy! >> firstfile
$ git commit -a -m ”Future conflict.”
(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8)-(9)-(11)-(12*) -- master
                                   \    /
                                    (10)-(13*) --mergebranch**

Quick back to master and add a line again to the firstfile. We really do want a conflict!

$ git checkout master
$ echo Conflict indeed! >> firstfile
$ git commit -a -m ”This will conflict.”

Use gitk to inspect. Use either ”gitk --all” to bring up a new instance, or just press ”ctrl+f5” on an existing.

(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8)-(9)-(11)-(12)-(14*) -- master**
                                   \    /
                                    (10)-(13*) --mergebranch

That just doesn't look that neat, straight forward anymore as did it when doing rebase, does it? But it keeps the history intact.

Now try to merge into master

$ git merge mergebranch

A conflict!
As usual we are overwhealmed.

Looking at the branchlist we see that we are still on the master branch. Since we are merging we are resolving the conflict throught a new merge commit.

Merge the firstfile manually as done previously or use git mergetool. This will bring up a merging tool, depending on what you have installed, and are able to specify via the --tools option. (kdiff is recommended on Linux, I use tortoisemerge on Windows)

$ git mergetool --tool tortoisemerge

Once resolved, commit the merge. This time git suggests a commit message itself.

(1)-(2)-(3)-(4)-(7)-(5)-(6)-(8)-(9)-(11)-(12)-(14)-(15*) -- master**
                                   \    /         /
                                    (10)-----(13*) --mergebranch

Other then rebase, merge allows you to keep the order of commits, but you do get an additional commit for each merge.
